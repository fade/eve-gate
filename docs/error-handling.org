* Error Handling Guide

Eve-gate uses the Common Lisp condition system for error handling, providing
a rich hierarchy of conditions for different error types.

[[file:index.org][Back to Index]] | [[file:api-reference.org][API Reference]]

** Condition Families at a Glance

Eve-gate exposes three distinct condition families, each tied to a
different lifecycle position:

| Family               | Package              | Lifecycle           | Raises as |
|----------------------+----------------------+---------------------+-----------|
| =esi-*= (wire-side)  | =eve-gate.core=      | Synchronous, raised | =error=   |
|                      |                      | from the HTTP path  |           |
| =scheduler-*=        | =eve-gate.concurrent= | Synchronous, raised | =error=   |
| (programmer error)   |                      | from =submit-refresh= |         |
|                      |                      | and =submit-bootstrap= |        |
| =esi-completion-event= | =eve-gate.concurrent= | Asynchronous,       | plain =condition= |
| (scheduler outcome)  |                      | delivered through   | — *not* an =error= |
|                      |                      | refresh callbacks   | subtype          |

The wire-side =esi-error= family is what =eve-gate.api:*= functions raise
when ESI returns a non-2xx response or the network fails. The scheduler
families are produced by the higher-level request-scheduler — see
[[file:concurrency.org#request-scheduler][Request Scheduler]] for context.

** Wire-side Hierarchy (=eve-gate.core=)

#+begin_example
esi-condition (base)
├── esi-deprecation-warning (warning)
└── esi-error (error base)
    ├── esi-client-error (4xx)
    │   ├── esi-bad-request (400)
    │   ├── esi-unauthorized (401)
    │   ├── esi-forbidden (403)
    │   ├── esi-not-found (404)
    │   └── esi-rate-limit-exceeded (420)
    ├── esi-server-error (5xx)
    │   ├── esi-internal-error (500)
    │   ├── esi-bad-gateway (502)
    │   ├── esi-service-unavailable (503)
    │   └── esi-gateway-timeout (504)
    └── esi-network-error
        ├── esi-connection-timeout
        └── esi-read-timeout
#+end_example

** Scheduler Programmer-Error Hierarchy (=eve-gate.concurrent=)

Raised synchronously by =submit-refresh= and =submit-bootstrap= when the
caller hands the scheduler something it cannot accept. Inherits from
=error=, catchable with =handler-case=.

#+begin_example
scheduler-error (error)
├── scheduler-missing-priority    ; :priority omitted on submit-refresh
├── scheduler-invalid-priority    ; :priority not in accepted set, incl. :bootstrap
├── scheduler-not-running         ; scheduler stopped or never started
├── scheduler-queue-full          ; queue would exceed its cap
└── scheduler-unknown-endpoint    ; operation-id not in find-endpoint
#+end_example

#+begin_src lisp
(handler-case
    (eve-gate.concurrent:submit-refresh *scheduler*
      :operation-id "get_markets_region_id_orders"
      :params '(:region-id 10000002 :type-id 587)
      ;; ... missing :priority intentionally
      )
  (eve-gate.concurrent:scheduler-missing-priority (c)
    (format t "Caller bug: priority not set for ~A~%"
            (eve-gate.concurrent:scheduler-error-operation-id c))))
#+end_src

These are programmer errors, not transient conditions — there is no
retry path. Fix the call site.

** Completion-Event Hierarchy (=eve-gate.concurrent=)

Delivered asynchronously through =refresh-completion='s =event= slot for
=:skipped= and =:failed= outcomes. The classes are plain conditions —
*not* =error= subtypes — because they are protocol-bearing data, not
unconditional faults. Consumer code that wants debugger landing escalates
explicitly via =(error event)=.

#+begin_example
esi-completion-event                   ; abstract — base for all
├── esi-skip                           ; abstract — no HTTP performed
│   ├── esi-budget-exhausted           ; held past deadline, budget shallow
│   ├── esi-deadline-missed            ; sat in engine queue past deadline
│   ├── esi-cancelled                  ; cancel-refresh / cancel-bootstrap
│   └── esi-queue-full                 ; submission expelled after acceptance
└── esi-failure                        ; abstract — HTTP attempted, failed
    ├── esi-http-error                 ; abstract — HTTP-level fault
    │   ├── esi-http-4xx-error
    │   └── esi-http-5xx-error
    ├── esi-network-failure            ; timeout, DNS, connection refused
    └── esi-rate-limit-exhausted       ; 420 retries exhausted
#+end_example

=esi-network-failure= is the scheduler-side class — the wire-side
=eve-gate.core:esi-network-error= is intentionally a different class so
that callers can distinguish "the network call failed while we were trying
to dispatch it" (wire-side) from "the scheduler is reporting that a
submission failed at the network layer" (scheduler-side). The scheduler
translates between them at its boundary.

*** Three ergonomic shapes for consumer code

**** A. Keyword switch on outcome (simplest)

#+begin_src lisp
(lambda (completion)
  (case (eve-gate.concurrent:refresh-completion-outcome completion)
    (:ok        (use-data (eve-gate.concurrent:refresh-completion-data completion)))
    (:cache-hit (use-cached (eve-gate.concurrent:refresh-completion-data completion)))
    (:skipped   (log-skip (eve-gate.concurrent:refresh-completion-event completion)))
    (:failed    (log-fail (eve-gate.concurrent:refresh-completion-event completion)))))
#+end_src

**** B. =typecase= on event class (family dispatch)

#+begin_src lisp
(lambda (completion)
  (typecase (eve-gate.concurrent:refresh-completion-event completion)
    (eve-gate.concurrent:esi-http-4xx-error      (handle-4xx (refresh-completion-event completion)))
    (eve-gate.concurrent:esi-http-5xx-error      (handle-5xx (refresh-completion-event completion)))
    (eve-gate.concurrent:esi-network-failure     (handle-net (refresh-completion-event completion)))
    (eve-gate.concurrent:esi-rate-limit-exhausted (handle-420 (refresh-completion-event completion)))
    (eve-gate.concurrent:esi-budget-exhausted    (defer-to-next-cycle))
    (eve-gate.concurrent:esi-deadline-missed     (log-late))
    (eve-gate.concurrent:esi-cancelled           nil)))
#+end_src

**** C. Re-signal for =handler-bind= (most powerful)

#+begin_src lisp
;; A refresh-cycle driver installs handlers; per-spec callbacks signal
;; the event so those handlers run with the right binding scope.
(handler-bind ((eve-gate.concurrent:esi-http-5xx-error  #'retry-or-log)
               (eve-gate.concurrent:esi-network-failure #'mark-stale-and-continue))
  (eve-gate.concurrent:submit-refresh-batch *scheduler* specs
    :on-batch-complete (lambda () (cycle-done))))

;; The wrapped per-spec callback signals the event:
(defun on-fail-handler (completion)
  (signal (eve-gate.concurrent:refresh-completion-event completion)))
#+end_src

=signal= is the lightweight half of the protocol — the condition need
not be raised as an error to be useful. =error= escalates to the
debugger; =signal= just runs the handlers and continues.

*** Common slot accessors

All =esi-completion-event= subclasses carry:

| Accessor                                       | Description                       |
|------------------------------------------------+-----------------------------------|
| =esi-completion-event-scheduler=               | the scheduler that produced it    |
| =esi-completion-event-handle=                  | the refresh-handle (back-ref)     |
| =esi-completion-event-operation-id=            | the operation-id that failed      |
| =esi-completion-event-endpoint=                | the resolved endpoint string      |
| =esi-completion-event-batch-id=                | pass-through batch-id if any      |
| =esi-completion-event-enqueued-at=             | universal-time of submission      |
| =esi-completion-event-completed-at=            | universal-time of outcome         |
| =esi-completion-event-wait-time=               | seconds in scheduler before outcome |

Subtype-specific slots add context — =esi-budget-exhausted= carries
=:limit-remain= and =:predicted-remain=; =esi-http-error= carries
=:status=, =:response-body=, =:response-headers=; =esi-network-failure=
wraps the underlying condition in =:cause=. See the source for the full
slot list.

** Basic Error Handling

*** Catching All ESI Errors

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id 123)
  (eve-gate.core:esi-error (c)
    (format t "ESI Error: ~A~%" c)))
#+end_src

*** Specific Error Handling

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id 1)
  (eve-gate.core:esi-not-found (c)
    (format t "Character not found: ~A~%" 
            (eve-gate.core:esi-error-message c)))
  (eve-gate.core:esi-rate-limit-exceeded (c)
    (format t "Rate limited. Retry after ~A seconds~%"
            (eve-gate.core:esi-rate-limit-retry-after c)))
  (eve-gate.core:esi-unauthorized (c)
    (format t "Authentication required or token expired~%"))
  (eve-gate.core:esi-error (c)
    (format t "Other ESI error (~A): ~A~%"
            (eve-gate.core:esi-error-status-code c)
            (eve-gate.core:esi-error-message c))))
#+end_src

** Error Condition Details

*** Common Accessors

All =esi-error= conditions provide:

#+begin_src lisp
(eve-gate.core:esi-error-message c)      ; Human-readable message
(eve-gate.core:esi-error-status-code c)  ; HTTP status code
(eve-gate.core:esi-error-endpoint c)     ; Endpoint that failed
(eve-gate.core:esi-error-request-id c)   ; ESI request ID for debugging
#+end_src

*** Client Errors (4xx)

**** 400 Bad Request

Invalid parameters or malformed request.

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id -1)
  (eve-gate.core:esi-bad-request (c)
    (format t "Invalid request: ~A~%" (eve-gate.core:esi-error-message c))))
#+end_src

**** 401 Unauthorized

Missing or invalid authentication.

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id-wallet *client* 
      :character-id 95465499)
  (eve-gate.core:esi-unauthorized (c)
    (format t "Need to authenticate first~%")
    ;; Trigger re-authentication flow
    ))
#+end_src

**** 403 Forbidden

Authenticated but lacking required scope or permission.

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id-wallet *client*
      :character-id 95465499)
  (eve-gate.core:esi-forbidden (c)
    (format t "Missing required scope: esi-wallet.read_character_wallet.v1~%")))
#+end_src

**** 404 Not Found

Resource doesn't exist.

#+begin_src lisp
(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id 1)
  (eve-gate.core:esi-not-found (c)
    (format t "Character ID 1 does not exist~%")))
#+end_src

**** 420 Rate Limited

#+begin_src lisp
(handler-case
    (make-many-requests)
  (eve-gate.core:esi-rate-limit-exceeded (c)
    (let ((retry-after (eve-gate.core:esi-rate-limit-retry-after c)))
      (format t "Rate limited. Waiting ~A seconds...~%" retry-after)
      (sleep retry-after)
      ;; Retry the request
      )))
#+end_src

*** Server Errors (5xx)

**** 500 Internal Server Error

#+begin_src lisp
(handler-case
    (eve-gate.api:get-status *client*)
  (eve-gate.core:esi-internal-error (c)
    (format t "ESI internal error, try again later~%")))
#+end_src

**** 502 Bad Gateway / 503 Service Unavailable / 504 Gateway Timeout

Usually transient - retry with backoff:

#+begin_src lisp
(handler-case
    (eve-gate.api:get-status *client*)
  (eve-gate.core:esi-server-error (c)
    (format t "Server error (~A), will retry~%"
            (eve-gate.core:esi-error-status-code c))
    (sleep 5)
    ;; Retry
    ))
#+end_src

*** Network Errors

#+begin_src lisp
(handler-case
    (eve-gate.api:get-status *client*)
  (eve-gate.core:esi-connection-timeout (c)
    (format t "Connection timed out~%"))
  (eve-gate.core:esi-read-timeout (c)
    (format t "Response timed out~%"))
  (eve-gate.core:esi-network-error (c)
    (format t "Network error: ~A~%" c)))
#+end_src

** Deprecation Warnings

ESI sometimes deprecates endpoints. Eve-gate signals warnings:

#+begin_src lisp
(handler-bind ((eve-gate.core:esi-deprecation-warning
                (lambda (w)
                  (format t "Deprecation: ~A~%" 
                          (eve-gate.core:esi-deprecation-message w))
                  (format t "Sunset date: ~A~%"
                          (eve-gate.core:esi-deprecation-sunset w))
                  (muffle-warning w))))
  (eve-gate.api:get-some-deprecated-endpoint *client*))
#+end_src

** Retry Patterns

*** Simple Retry

#+begin_src lisp
(defun with-retry (fn &key (max-attempts 3) (delay 1))
  "Retry FN up to MAX-ATTEMPTS times with DELAY between attempts."
  (loop for attempt from 1 to max-attempts
        do (handler-case
               (return (funcall fn))
             (eve-gate.core:esi-server-error (c)
               (when (= attempt max-attempts)
                 (error c))
               (format t "Attempt ~A failed, retrying...~%" attempt)
               (sleep delay)))))
#+end_src

*** Exponential Backoff

#+begin_src lisp
(defun with-exponential-backoff (fn &key (max-attempts 5) (initial-delay 1))
  "Retry FN with exponential backoff."
  (loop for attempt from 1 to max-attempts
        for delay = initial-delay then (* delay 2)
        do (handler-case
               (return (funcall fn))
             (eve-gate.core:esi-error (c)
               (when (= attempt max-attempts)
                 (error c))
               (format t "Attempt ~A failed, waiting ~As...~%" attempt delay)
               (sleep delay)))))
#+end_src

*** Rate Limit Aware Retry

#+begin_src lisp
(defun rate-limit-aware-call (fn)
  "Call FN, respecting rate limit retry-after headers."
  (handler-case
      (funcall fn)
    (eve-gate.core:esi-rate-limit-exceeded (c)
      (let ((retry-after (or (eve-gate.core:esi-rate-limit-retry-after c) 5)))
        (format t "Rate limited, waiting ~As~%" retry-after)
        (sleep retry-after)
        (funcall fn)))))
#+end_src

** Error Logging

#+begin_src lisp
(defun log-error (condition)
  "Log an ESI error with relevant details."
  (eve-gate.utils:log-error 
   "ESI request failed"
   :status-code (eve-gate.core:esi-error-status-code condition)
   :endpoint (eve-gate.core:esi-error-endpoint condition)
   :request-id (eve-gate.core:esi-error-request-id condition)
   :message (eve-gate.core:esi-error-message condition)))

(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id 123)
  (eve-gate.core:esi-error (c)
    (log-error c)
    (error c)))
#+end_src

** Best Practices

*** Be Specific

Handle specific conditions when you can take specific action:

#+begin_src lisp
;; Good: handle rate limiting specially
(handler-case
    (fetch-data)
  (eve-gate.core:esi-rate-limit-exceeded (c)
    (sleep (eve-gate.core:esi-rate-limit-retry-after c))
    (fetch-data))
  (eve-gate.core:esi-error (c)
    (log-and-report c)))
#+end_src

*** Don't Swallow Errors

#+begin_src lisp
;; Bad: silently ignores errors
(handler-case (fetch-data)
  (error () nil))

;; Good: log and re-raise or handle appropriately
(handler-case (fetch-data)
  (eve-gate.core:esi-not-found ()
    nil)  ; OK to return nil for "not found"
  (eve-gate.core:esi-error (c)
    (log-error c)
    (error c)))  ; Re-raise unexpected errors
#+end_src

*** Use Request IDs

ESI provides request IDs for debugging. Include them in bug reports:

#+begin_src lisp
(handler-case
    (fetch-data)
  (eve-gate.core:esi-error (c)
    (format t "Error! Request ID: ~A~%" 
            (eve-gate.core:esi-error-request-id c))
    (format t "Please report this ID to CCP if the error persists~%")))
#+end_src
