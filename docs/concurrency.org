* Concurrency Guide

Eve-gate provides rate limiting and parallel request capabilities to help
you make efficient use of the ESI API while respecting its limits.

[[file:index.org][Back to Index]] | [[file:api-reference.org][API Reference]]

** ESI Rate Limits

The EVE Swagger Interface has rate limits to prevent abuse:

| Limit Type    | Rate                    | Description                       |
|---------------+-------------------------+-----------------------------------|
| Global        | ~150 requests/second    | Per-IP address across all apps    |
| Error         | Cumulative              | Errors count more heavily         |
| Per-endpoint  | Varies                  | Some endpoints have lower limits  |
| Per-character | ~20/second              | Authenticated endpoints           |

Eve-gate automatically handles these limits using token bucket rate limiters.

** Rate Limiting

*** Automatic Rate Limiting

Rate limiting is enabled by default:

#+begin_src lisp
(defvar *client* (eve-gate:make-eve-client))

;; Requests are automatically throttled to stay within limits
(loop for id in character-ids
      collect (eve-gate.api:get-characters-character-id *client* 
                                                         :character-id id))
#+end_src

*** Token Bucket Algorithm

Eve-gate uses the token bucket algorithm:

- Buckets hold tokens (representing allowed requests)
- Tokens are consumed when making requests
- Tokens refill at a constant rate
- If no tokens available, request waits

#+begin_src lisp
;; Create a custom rate limiter
(defvar *limiter*
  (eve-gate.concurrent:make-token-bucket 
   :max-tokens 100      ; Burst capacity
   :refill-rate 10))    ; Tokens per second

;; Try to acquire a token
(eve-gate.concurrent:bucket-try-acquire *limiter*)
;; => T (if token available)
;; => NIL (if no tokens)

;; Check available tokens
(eve-gate.concurrent:bucket-tokens-available *limiter*)
;; => 99.5
#+end_src

*** Per-Endpoint Limits

Some endpoints have stricter limits:

#+begin_src lisp
;; Search endpoints are heavily limited
(eve-gate.api:get-characters-character-id-search *client*
  :character-id 95465499
  :search "rifter"
  :categories '("inventory_type"))
;; Rate limited to ~10/minute per character

;; Market endpoints have moderate limits
(eve-gate.api:get-markets-region-id-orders *client*
  :region-id 10000002
  :type-id 587)
;; Rate limited but higher than search
#+end_src

** Bulk Operations

### Parallel Requests

For fetching multiple items efficiently:

#+begin_src lisp
;; Fetch multiple characters in parallel
(eve-gate.concurrent:bulk-get *client*
  '("/characters/95465499/"
    "/characters/95465500/"
    "/characters/95465501/"))
;; => List of responses (in order)

;; With custom concurrency
(eve-gate.concurrent:bulk-get *client* urls
  :max-concurrent 10    ; Parallel workers
  :timeout 30)          ; Per-request timeout
#+end_src

*** Bulk Operations

#+begin_src lisp
;; General bulk operation
(eve-gate.concurrent:bulk-operations *client*
  (list 
   (lambda (c) (eve-gate.api:get-characters-character-id c :character-id 123))
   (lambda (c) (eve-gate.api:get-characters-character-id c :character-id 456))
   (lambda (c) (eve-gate.api:get-corporations-corporation-id c :corporation-id 789)))
  :max-concurrent 5)
#+end_src

*** Paginated Fetching

#+begin_src lisp
(defun fetch-all-pages (client endpoint-fn &rest args)
  "Fetch all pages of a paginated endpoint."
  (multiple-value-bind (first-page response)
      (apply endpoint-fn client :page 1 args)
    (let ((total-pages (eve-gate.core:esi-response-pages response)))
      (if (<= total-pages 1)
          first-page
          (append first-page
                  (loop for page from 2 to total-pages
                        append (apply endpoint-fn client :page page args)))))))

;; Usage
(fetch-all-pages *client* #'eve-gate.api:get-universe-types)
#+end_src

### Queue Management

Eve-gate queues requests when rate limited:

#+begin_src lisp
;; Check queue status
(eve-gate.concurrent:queue-status *request-queue*)

;; Queue metrics
(eve-gate.concurrent:queue-pending-count *request-queue*)
(eve-gate.concurrent:queue-processed-count *request-queue*)
#+end_src

*** Timeout Handling

#+begin_src lisp
;; Configure queue timeout
(defvar *client*
  (eve-gate:make-eve-client
   :config '(:queue-timeout 60)))  ; Wait up to 60s for rate limit

;; Handle timeout
(handler-case
    (eve-gate.api:get-characters-character-id *client* :character-id 123)
  (eve-gate.core:esi-rate-limit-exceeded (c)
    (format t "Rate limited, retry after ~A seconds~%" 
            (eve-gate.core:esi-rate-limit-retry-after c))))
#+end_src

** Concurrent Patterns

*** Worker Pool

#+begin_src lisp
(defun process-characters (client character-ids processor-fn)
  "Process characters in parallel with controlled concurrency."
  (let ((results (make-array (length character-ids) :fill-pointer 0)))
    (eve-gate.concurrent:with-worker-pool (pool :workers 5)
      (loop for id in character-ids
            do (eve-gate.concurrent:submit-job pool
                 (lambda ()
                   (let ((data (eve-gate.api:get-characters-character-id 
                                client :character-id id)))
                     (vector-push-extend (funcall processor-fn data) results))))))
    results))
#+end_src

*** Batch Processing

#+begin_src lisp
(defun batch-process (items batch-size process-fn)
  "Process items in batches with delays between batches."
  (loop for batch in (partition-all batch-size items)
        do (mapc process-fn batch)
           (sleep 1)))  ; Rate limit protection

;; Usage
(batch-process character-ids 10
  (lambda (id) 
    (eve-gate.api:get-characters-character-id *client* :character-id id)))
#+end_src

** Request Scheduler

The request-scheduler is the high-level submission surface above the engine,
rate-limiter, and cache-manager. It coordinates three things that the lower
primitives do not compose on their own: cache short-circuit at submission
time, predicted-budget gating that pauses non-critical dispatch when ESI's
error budget is shallow, and a paced bulk-drain channel for cold-start
fetches that should not compete with the steady-state refresh cadence.

Returns =refresh-handle= objects for individual submissions and
=bootstrap-handle= objects for batches. Both carry the eventual outcome
through typed callbacks, so consumer code never needs to poll.

*** When to reach for the scheduler

- *Heat-map refresh driver.* The consumer code is selecting endpoints from
  a volatility heat-map and dispatching at a tiered cadence. The scheduler
  accepts those submissions with =:priority :critical / :hot / :warm / :cold=,
  short-circuits any whose cache entry is still fresh, and defers
  non-critical work when the budget gate closes.
- *Universe-mirror seed.* A one-shot drain of tens of thousands of
  orderbook fetches needs to run at ESI's safe bulk rate (~5 req/sec) for
  several hours without starving the steady-state heat-map. =submit-bootstrap=
  routes those onto a paced channel with deadline-tolerant 6-hour expiry.
- *Operator-facing dashboards.* The scheduler exposes a single =scheduler-state=
  plist covering budget, queue, bootstrap, engine, cache, and recent
  failures — designed to be polled at 1Hz by a monitoring harness.

When the caller just needs one-off requests with caching and rate-limiting,
=make-eve-client= already wires the production HTTP stack and the higher-level
=eve-gate.api:*= verbs are enough. Reach for =request-scheduler= when the
caller needs the gate, bulk channel, or unified observation.

*** Lifecycle

#+begin_src lisp
;; Construct: cache-manager is optional but recommended.
(defvar *scheduler*
  (eve-gate.concurrent:make-request-scheduler
   :cache-manager (eve-gate.cache:make-cache-manager)))

;; Start the dispatcher and engine workers.
(eve-gate.concurrent:start-scheduler *scheduler*)

;; ... submissions ...

;; Stop cleanly.
(eve-gate.concurrent:stop-scheduler *scheduler* :wait t)
#+end_src

The =with-scheduler= macro handles construction, start, and unwind-safe stop
in a single form:

#+begin_src lisp
(eve-gate.concurrent:with-scheduler (s :cache-manager cm)
  (eve-gate.concurrent:submit-refresh s
    :operation-id "get_status"
    :priority :warm
    :on-complete (lambda (c) (process-status c))))
#+end_src

Both =start-scheduler= and =stop-scheduler= are idempotent.

*** Submitting a refresh

=submit-refresh= takes an =:operation-id= (the ESI swagger ID as a string or
symbol — both =get-status= and ="get_status"= resolve to the same endpoint),
any path or query =:params= as a plist, a required =:priority=, and up to
three callbacks. Returns a =refresh-handle=.

#+begin_src lisp
(eve-gate.concurrent:submit-refresh *scheduler*
  :operation-id "get_markets_region_id_orders"
  :params '(:region-id 10000002 :type-id 587 :order-type "sell")
  :priority :hot
  :deadline-seconds 300
  :batch-id "heat-map-cycle-42"
  :on-complete (lambda (completion)
                 (process-orders (eve-gate.concurrent:refresh-completion-data completion)))
  :on-skip (lambda (completion)
             (log-skip (eve-gate.concurrent:refresh-completion-event completion)))
  :on-fail (lambda (completion)
             (log-failure (eve-gate.concurrent:refresh-completion-event completion))))
#+end_src

*Priority is required.* The scheduler refuses to guess a tier — omission
signals =scheduler-missing-priority=. Accepted values:

| Priority    | Engine class       | Default deadline | Use case                              |
|-------------+--------------------+------------------+---------------------------------------|
| =:critical= | error recovery     | 30 s             | Auth flows, error-handling re-fetches |
| =:hot=      | user-initiated     | 5 min            | Heat-map's volatile-endpoint cadence  |
| =:warm=     | standard refresh   | 30 min           | Heat-map's moderate cadence           |
| =:cold=     | background prefetch | 6 h              | Periodic refresh of stable data       |
| =:bootstrap= | (internal)         | 6 h              | Use =submit-bootstrap= — see below     |

=:bootstrap= as a =:priority= argument to =submit-refresh= signals
=scheduler-invalid-priority=. Bulk drains have their own verb because they
share the engine but use a separately-paced dispatcher branch.

*** Outcomes and the completion struct

The three callbacks correspond to three terminal outcomes:

| Callback     | Fires for outcomes  | =event= slot value         |
|--------------+---------------------+----------------------------|
| =:on-complete= | =:ok=, =:cache-hit= | nil                        |
| =:on-skip=     | =:skipped=          | =esi-skip= instance        |
| =:on-fail=     | =:failed=           | =esi-failure= instance     |

Every callback receives a =refresh-completion= struct with timing data,
the endpoint and operation-id, the batch-id, the attempt count, and either
the response data (=:ok=, =:cache-hit=) or a typed condition instance carrying
the reason (=:skipped=, =:failed=). The condition hierarchy is detailed in the
[[file:error-handling.org][Error Handling Guide]].

#+begin_src lisp
;; Pattern: branch on outcome, then on event class for the exceptional cases.
(lambda (completion)
  (case (eve-gate.concurrent:refresh-completion-outcome completion)
    (:ok        (use-fresh-data (eve-gate.concurrent:refresh-completion-data completion)))
    (:cache-hit (use-cached-data (eve-gate.concurrent:refresh-completion-data completion)))
    (:skipped   (typecase (eve-gate.concurrent:refresh-completion-event completion)
                  (eve-gate.concurrent:esi-budget-exhausted (defer-to-next-cycle))
                  (eve-gate.concurrent:esi-deadline-missed  (log-late))
                  (eve-gate.concurrent:esi-cancelled        (no-op))))
    (:failed    (handle-failure (eve-gate.concurrent:refresh-completion-event completion)))))
#+end_src

*** Cache short-circuit

When the scheduler has a =cache-manager= attached and the submission's
(endpoint, params, auth-context) tuple is fresh in the memory tier, the
completion is delivered synchronously with =outcome :cache-hit=. No
rate-limit token is consumed and no engine submission happens — the
callback fires inside =submit-refresh= before the call returns.

Stale entries with a stored ETag fall through to the normal dispatch
path, where the request-pipeline's cache middleware adds the
=If-None-Match= header. ESI returns 304 if unchanged, and the response
middleware serves the cached body. The scheduler does not add
=If-None-Match= itself — that responsibility lives on the middleware,
which prevents two competing conditional-request headers from landing on
the wire.

*** Predicted-budget gate

The scheduler keeps an internal /in-flight/ counter — submissions
that have been dispatched but not yet completed — and subtracts it
(plus a small jitter margin) from the rate-limiter's
=X-ESI-Error-Limit-Remain= snapshot. The result is
=predicted-remain=: the budget the scheduler thinks will be left
after the currently-flying requests land.

When =predicted-remain= drops below =budget-threshold= (default 30), the
gate closes and non-critical submissions park in a held-queue indexed by
engine priority instead of dispatching. When =predicted-remain= climbs
back above =budget-resume= (default 50) the gate reopens. =:critical=
always bypasses the gate.

The gate's hysteresis prevents flap: the threshold to close (30) is
lower than the threshold to reopen (50), so a budget that hovers around
the line does not toggle the gate every poll cycle.

Held submissions whose deadline arrives before the gate reopens are
delivered as =:skipped= with an =esi-budget-exhausted= event.

#+begin_src lisp
;; The dispatcher reacts to ESI's actual error-limit headers — there is no
;; need to manually configure the gate's thresholds for normal operation.
;; Tuning is offered when the consumer's traffic shape is unusual:
(eve-gate.concurrent:make-request-scheduler
  :budget-threshold 50    ; close earlier
  :budget-resume    100   ; reopen later
  :jitter-margin    10)   ; conservative bias under bursty traffic
#+end_src

*** Bulk drain — =submit-bootstrap=

For cold-start fetches that need to run at ESI's safe bulk rate without
competing with the steady-state heat-map. =submit-bootstrap= accepts a
list of spec plists (each with =:operation-id=, =:params=, and per-spec
=:on-complete= / =:on-skip= / =:on-fail=), parks them on a held-queue
slot reserved for bulk priority, and drains them paced by a per-scheduler
token bucket (default 5 req/sec).

Returns a =bootstrap-handle= aggregating the per-spec handles plus
counters for settled, ok, failed, and skipped specs.

#+begin_src lisp
;; Seed a region's orderbook — every market type in the region.
(let ((b (eve-gate.concurrent:submit-bootstrap *scheduler*
           (loop for type-id in market-type-ids
                 collect (list :operation-id "get_markets_region_id_orders"
                               :params (list :region-id 10000002
                                             :type-id type-id)
                               :on-complete (lambda (c) (insert-orders c))))
           :batch-id "the-forge-seed"
           :on-progress (lambda (settled total)
                          (format t "~D / ~D specs settled~%" settled total))
           :on-complete (lambda (b-handle)
                          (format t "Seed done: ~D ok, ~D failed, ~D skipped~%"
                                  (eve-gate.concurrent:bootstrap-handle-ok-count b-handle)
                                  (eve-gate.concurrent:bootstrap-handle-failed-count b-handle)
                                  (eve-gate.concurrent:bootstrap-handle-skipped-count b-handle))))))
  (use-bootstrap-handle b))
#+end_src

Pacing is the binding constraint, not queue-walking — the dispatcher
drains the bootstrap queue from the head only and stops as soon as the
head is non-dispatchable, so per-tick cost is O(K) where K is the tokens
acquired, independent of the queue's depth. A 70K-spec batch costs the
same per tick as a 5-spec batch.

Cache-fresh specs short-circuit synchronously and do not consume pacing
tokens — a partially-warm seed only pays for the misses.

Bulk submissions past =bootstrap-queue-depth= (default 80000) signal
=scheduler-queue-full= at the API boundary.

*** Cancellation

#+begin_src lisp
;; Cancel a single still-pending refresh.
(eve-gate.concurrent:cancel-refresh *scheduler* handle)
;; => T if the handle was :pending and got cancelled
;; => NIL if already :dispatched, :completed, or :cancelled

;; Cancel every still-pending spec in a bootstrap batch.
(eve-gate.concurrent:cancel-bootstrap *scheduler* bootstrap-handle)
;; => integer count of refresh-handles actually cancelled
#+end_src

A cancelled pending handle delivers a =:skipped= completion with an
=esi-cancelled= event through the consumer's =:on-skip= callback. Already
=:dispatched= handles complete naturally with their eventual response —
cancellation is honest, not surprising. There is no way to abort an
in-flight HTTP request without confusing the consumer's bookkeeping.

*** Observation — =scheduler-state=

A single call returns a multi-block plist designed for sustained 1Hz
polling by a monitoring harness. Each sub-block acquires only its
subsystem's locks; the snapshot is not atomic across subsystems but
consistent within each one.

#+begin_src lisp
(eve-gate.concurrent:scheduler-state *scheduler*)
;; => (:budget    (:limit-remain 87 :limit-reset-seconds 42
;;                 :backoff-remaining 0.0 :consecutive-420s 0
;;                 :in-flight 3 :predicted-remain 79)
;;     :queue     (:current-depth 2 :depths-by-priority #(0 1 1 0 0)
;;                 :total-enqueued 1234 :total-dequeued 1232 ...)
;;     :bootstrap (:active-p T :batch-id "the-forge-seed"
;;                 :pending 6843 :completed 12 :failed 0 :paced-rate 5.0)
;;     :engine    (:total-requests 1232 :success-rate 0.998
;;                 :avg-latency 0.187 :throughput 4.3 :worker-count 8)
;;     :cache     (:hit-rate 0.42 :l1-hits 521 :l2-hits 0 :etag-hits 39 ...)
;;     :scheduler (:total-submitted 1234 :total-cache-hits 521
;;                 :total-dispatched 700 :total-skipped 13 :total-failed 0
;;                 :skip-reasons (:cancelled 8 :budget-exhausted 5))
;;     :recent-failures (...))
#+end_src

The =:bootstrap= block is NIL when no batch is registered. The =:cache=
block is NIL when no cache-manager is attached.

=reset-scheduler-stats= zeros the cumulative scheduler-private counters
without touching live state (in-flight, active handles, or registered
bootstrap batches). Subsystem-specific stats (rate-limiter, engine,
queue, cache) are reset separately via their own reset verbs.

** Best Practices

*** Minimize Requests

- Use bulk endpoints when available (e.g., =post-characters-affiliation=)
- Cache responses appropriately (see [[file:caching.org][Caching]])
- Batch related operations

*** Handle Rate Limits Gracefully

#+begin_src lisp
(defun resilient-fetch (client fetch-fn &key (max-retries 3))
  "Fetch with automatic retry on rate limiting."
  (loop for attempt from 1 to max-retries
        do (handler-case
               (return (funcall fetch-fn client))
             (eve-gate.core:esi-rate-limit-exceeded (c)
               (when (< attempt max-retries)
                 (sleep (or (eve-gate.core:esi-rate-limit-retry-after c) 5)))))))
#+end_src

*** Respect Error Limits

ESI tracks error rates. Too many errors can result in temporary bans:

#+begin_src lisp
;; Check error window header
(eve-gate.core:esi-response-error-limit-remain response)
(eve-gate.core:esi-response-error-limit-reset response)

;; Back off when errors accumulate
(when (< error-limit-remain 10)
  (sleep error-limit-reset))
#+end_src
